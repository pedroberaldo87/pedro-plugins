## Context

**Problema observado (Pedro, premissa da sessão):** num codebase grande, o project-doc "tira o pé". Em vez de minerar de verdade (rastreio do código + reconciliação finding-por-finding, a proposta original), o agente single-window se poupa sob o volume de contexto e segue a documentação que já existe. A doc resultante é rasa — a mineração full nunca acontece.

**Causa-raiz (confirmada no código):** a COLETA é Python determinístico (`journal.py`/`collect_engine.py`, tiers 2-4) e não tira o pé. Quem tira o pé é o **agente na janela**, em dois pontos que escalam com o tamanho do codebase: (a) o Tier 1 scan do código real (`SKILL.md:121-125`, regra "no shortcuts") e (b) a Projeção/Reconciliação obrigatória (`SKILL.md:126`, confirmar cada finding contra o código, tratar os `stale_ids`). Numa janela só, sob volume, ambos são cortados.

**Solução:** o padrão de documentação full passa a rodar via **Workflow** (orquestração multi-agente determinística). Cada agente recebe **uma fatia** (um concern/doc) → working-set pequeno → ninguém tem "medo de contexto" → a soma cobre tudo. Mais: quando já existe doc, uma sequência **backup → minera do zero → compara → preserva nuances** garante o melhor dos dois mundos (mineração fresca + curadoria que só vivia na doc antiga). E uma **checagem ativa** detecta doc fora do padrão atual e força essa sequência automaticamente.

**Intended outcome:** `/project-doc` full nunca mais entrega doc rasa por preguiça de contexto; doc antiga/fora-do-padrão é reconstruída por mineração sem perder nuance; o custo extra fica isolado nos modos que mineram (FULL/`--deep`).

---

## Decisões (ratificadas)

1. **Gatilho do workflow = por MODO.** FULL e `--deep` → workflow, sempre, independente de tamanho. Modos leves (incremental de 1 doc, `index`, `pointers`, `--rebuild`, `migrate`, `verify`, `clean`) → single-agent (não mineram; nada a paralelizar). Detecção de escala serve só pra **dimensionar** o fan-out, não pra decidir se usa.
2. **Reintegração de nuances = automática.** O garimpeiro extrai as nuances ausentes na doc nova, valida cada uma contra o código atual; as **confirmadas** são reintegradas sozinhas (via journal), as **não-confirmáveis** entram `[relatado]`, as **contraditas** viram `invalidate`. Pedro revê tudo no relatório final — sem checkpoint humano no meio.
3. **Checagem ativa (novo requisito).** No passo 0, classificar a doc existente; se estiver **fora do padrão atual**, escalonar automaticamente para a sequência full (backup + mineração deep + compare), em vez de um update delta leve.

---

## Arquitetura

### A. Fronteira de modos

| Modo | Motor | Mineração |
|---|---|---|
| FULL (`/project-doc`) | **Workflow** | `journal.py update` (delta) ou `deep` (se checagem ativa mandar) |
| `--deep` | **Workflow** | `journal.py deep` (cold-start, todas as sessões) |
| incremental / index / pointers / `--rebuild` / migrate / verify / clean | single-agent (como hoje) | delta/nenhuma |

Flag de escape opcional `--solo` força single-agent num full (debug). Não anuncia custo nem pede confirmação — dispara direto (Pedro: "dane-se").

### B. Checagem ativa do estado da doc (passo 0.1 — NOVO)

Classifica antes de decidir o modo:

- **Ausente** (sem `.claude/CLAUDE.md`) → CREATE: workflow + `deep` (cold-start). Sem backup/compare (não há antiga).
- **Fora do padrão** → **sequência full forçada** (backup + workflow + `deep` + compare). É "fora do padrão" se QUALQUER:
  - markers v1 (`project-doc:start/end`), ou sem markers, ou doc escrita à mão;
  - markers v2 **mas sem journal** (`.claude/.project-doc/findings.jsonl` ausente) → doc gerada por motor pré-v3, nunca minerada — o caso que mais engana;
  - marker registra `gen=<versão>` inferior à versão atual do plugin (motor mudou de padrão).
- **No padrão** (markers v2 + journal presente + `gen` atual) → FULL normal: workflow + `update` (delta) + backup/compare (sempre que há doc, pra preservar nuance).

Para suportar a 3ª regra, o marker da doc passa a registrar a versão do gerador: `<!-- project-doc:v2 gen=3.1.0 -->`. Sem isso, "fora do padrão por motor antigo" é indetectável quando o formato (v2) não muda.

### C. O fluxo full via Workflow

Divisão **casca (skill) ↔ workflow (motor)**, espelhando o molde do qa-loop (`plugins/qa-loop/skills/qa-loop/SKILL.md:22-38, 142-212`): casca faz o que o Workflow não consegue (decisão, dimensionamento, persistência, ratificação); o Workflow roda em background, recebe tudo via `args`, gates são JS.

**CASCA — passo 0 (antes):**
1. Identify root/layout/type/PM + **checagem ativa (B)** → decide `update` vs `deep` e se força a sequência.
2. Backup (se há doc) → `.claude/.project-doc/backups/<UTC-ts>/` (CLAUDE.md + docs/ + MANIFEST.json). `cp` puro, sem agente.
3. Roda a COLETA Python **1×** (`journal.py update|deep`) → `{live[], stale_ids, ...}` (`journal.py:774`). A mineração nunca entra no fan-out — é determinística e barata.
4. **Dimensiona** o fan-out e particiona `live[]`/`stale_ids` por concern (roteamento grosso por `raw_kind`+anchor).

**WORKFLOW — fases:**
- **Fase A — SCAN+RECONCILE (paralelo):** 1 agente por concern (+ 1 por `app×concern` no monorepo). Cada agente: lê os arquivos-fonte reais do seu concern (mata o ponto a), reconcilia os findings do seu concern contra o código que leu (mata o ponto b), escreve a seção. Devolve `DOC_SECTION`. **A doc nova é a BASE canônica — projetada SEM ver a doc antiga.**
- **Fase B — GARIMPEIRO DE NUANCES (1 agente, só se há doc antiga):** recebe a doc antiga (backup) + a nova + acesso de leitura ao código/journal. Tarefa **negativa**: achar info verdadeira presente só na antiga e **ausente** na nova; validar cada candidata contra o código. Devolve `NUANCE_CANDIDATES`. Travas anti-viés: a nova já existe e é base; o agente só pode **propor adições** (schema), nunca reescrever; conteúdo presente em ambas é descartado por construção.
- **Fase C — STITCH (JS puro, sem LLM):** aplica gates (invalidação, dedup, inline-vs-doc, secret CRITICAL, token budget, cobertura), monta índice. Retorna `STITCH_RESULT`.

**CASCA — passo final (depois):**
5. Aplica no journal **só a casca** (serializa a escrita append-only): invalidações aprovadas + nuances confirmadas (`curate`/`adopt`/`invalidate`).
6. **Re-projeta** (`journal.py rebuild`) pra materializar as nuances pela porta canônica — assim sobrevivem a toda rodada futura (sem isso, o próximo `--rebuild` apagaria edição manual na doc; `SKILL.md` Update Mechanism).
7. Escreve arquivos + verification (inclui check #10 secret) + relatório com telemetria (nº de agentes, invalidações aplicadas vs propostas, nuances reintegradas).

### D. Peça nova no motor (única adição ao Python)

`curate` e `invalidate` exigem id pré-existente (`journal.py:800-801, 815-816`) — não há como injetar uma nuance da doc antiga que **nunca foi minerada**. Adicionar subcomando **`journal.py adopt --text "..." --raw-kind <k>`**: cria um evento `discovered` com `source.type="curated_from_doc"`, id estável via `finding_id()`, passando pelo **mesmo scrubber** (`journal.py:737`). Reusa `_candidate()` (`journal.py:563`) + `append_events()` (`journal.py:113`). Simétrico aos coletores existentes; é a forma honesta (a nuance vira finding de 1ª classe, viaja no `findings.jsonl` versionado, sobrevive a `--rebuild`).

### E. Schemas (campos pros gates lerem, não texto solto)

- **`DOC_SECTION`**: `{concern, app, complete, doc_path, inline, body_md, confirmed_ids[], relatado_ids[], invalidations[{id,reason,evidence,confidence}], nuances[], todos[], secret_suspects[]}`
- **`NUANCE_CANDIDATES`**: `{candidates[{type, claim, where_in_old, validation{status,evidence}, match_to_journal{finding_id,relation}, proposed_action}], summary}`
- **`STITCH_RESULT`**: `{index_md, docs_to_write[], inline_sections[], approved_invalidations[], rejected_invalidations[], dedup_log[]}`

Gate crítico de invalidação: o LLM **propõe**, o JS **adjudica** (só aplica se `confidence=high` + `evidence` não-vazio + id no `live[]`). Invalidar é destrutivo no journal — low-confidence vira `[relatado]`, não morte.
