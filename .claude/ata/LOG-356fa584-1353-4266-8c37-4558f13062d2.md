# project-doc v3 — documentação como projeção de um journal de evidência

## Context

Hoje o **project-doc** (v2.2.1, 1056 linhas, 100% prosa) é um **file-scanner cego**: a Detection Matrix só lê arquivos do disco. Gotachas, decisões e features que nasceram em sessões de trabalho — o conhecimento mais valioso — só entram na doc se alguém já os escreveu à mão num ADR/README. Ex.: o gotcha *"hook de plugin vai em hooks/hooks.json, NUNCA na raiz"* nasceu de uma sessão de debugging; o project-doc não o descobriria sozinho.

Pedro: *"documentação é documentação, tem que ter TODAS as informações. Não tem cabimento você tateando pra adivinhar os problemas."* O objetivo desta mudança é parar de tatear: usar **toda** evidência que o projeto tem (arquivos, conversas, commits, handoffs, memória, grafo), e materializar isso numa doc **completa e auto-limpante**, sem perder nada e sem vazar secret.

Caso real de escala: o **VIUSTUDIO-TOOLS** tem **303 sessões / ~270 MB** de transcript espalhadas em **11 slugs** de `~/.claude/projects` (um por CWD).

## A arquitetura (decisões fechadas com Pedro)

### 1. Fontes em cascata de 5 tiers
Detecção deixa de ser só "arquivos". Por ordem de densidade/custo:
- **Tier 1 — o que existe** (já faz): scan de arquivo (stack, deps, rotas, schema).
- **Tier 2 — destilado pronto** (novo, barato): lê **HANDOFF*.md**, **memory/*.md**, e os `manifest-*.json`/`LOG-*.md` das atas; e **lê o grafo** graphify (god nodes → arquitetura, comunidades → módulos) em vez de só sugerir criar.
- **Tier 3 — histórico com intenção** (novo): mensagens de commit (`git log`) — o "porquê" das mudanças.
- **Tier 4 — mineração verbatim** (novo, caro): transcripts `.jsonl` via a engine de extração.
- **Tier 5 — o humano** (novo, último recurso): lacuna crítica sem fonte → pergunta, em vez de marcar `[TODO]` e seguir.

### 2. Journal + projeção (event sourcing — o "super git")
- **findings.jsonl = log de eventos append-only**, versionado no git. Nunca deleta — igual git nunca apaga commit. É a fonte de verdade histórica.
- **Doc canônica (CLAUDE.md + .claude/docs) = projeção do estado vivo.** Só o que vale agora; limpa; re-derivável do journal a qualquer momento (como o working tree do git).

### 3. Delta inteligente — 2 direções
- **Forward** (novo): sessões/commits novos → minera → eventos `discovered` no journal.
- **Backward** (invalidação): cada finding tem **âncoras** (paths que toca, via `collect_edited_paths`). `git diff last_commit..HEAD` → arquivos mudados → re-valida **só** os findings ancorados neles. Se um commit contradiz um gotcha, registra evento `invalidated` (não deleta). A projeção exclui findings invalidados → **a doc auto-limpa sem o journal perder nada.**
- **Ledger** (versionado) rastreia o consumido: `mined_sessions`, `last_commit`, `distilled_hashes`. Sessões fechadas mineradas 1×; a sessão ativa sempre (cresce).

### 4. Privacidade — findings versionado + scrubber
- O findings **vai pro git** (revertido de "fora do git"): os transcripts são locais por máquina e **não viajam**; o findings versionado é o único veículo do conhecimento entre máquinas/clones.
- **Scrubber** roda na escrita do journal: política **"nomes e contexto sim, valores não"** (a regra de ouro que a skill já aplica na doc, estendida ao journal). Preserva nome de var, host, porta, path; redige só o valor-secreto. Hosts/IPs internos: **preserva** (contexto de infra). Na dúvida: **preserva e marca** `‹revisar?›`.

### 5. Cofre Operacional — o scrubber MOVE o secret, não deleta
- Ao achar `SSH_HOST=191.x…`, grava o **valor** no cofre local **e** deixa `SSH_HOST → ‹cofre›` na doc/journal. Nada se perde.
- **Cofre real** em `iCloud Drive/Cofre/<projeto>.env` (sincroniza entre máquinas sozinho); cada repo tem `.claude/secrets/ops.env` como **link gitignored** apontando pra lá.
- A doc **referencia** o cofre, pra que numa sessão zerada o agente leia a doc → ache a referência → abra o cofre → tenha host/chave pra fazer deploy/SSH sem o Pedro relembrar.

### 6. Engine compartilhada via vendoring no monorepo
- O Claude Code **isola plugins na instalação** (só `plugins/<nome>/` vai pro cache; sem variável cross-plugin) → compartilhar código em runtime é inviável.
- Solução: **fonte-da-verdade única no source + vendoring no build.** A engine de extração vira asset neutro em **_shared/**; um **scripts/sync-shared.sh** copia pra dentro de cada plugin que usa, antes de commitar. Source DRY, plugins autônomos no runtime.
