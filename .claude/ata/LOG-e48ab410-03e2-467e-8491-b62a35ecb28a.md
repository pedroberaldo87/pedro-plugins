## A estrutura, explicada

```
plugins/guardrails/                 ← a pasta = 1 plugin (o nome da pasta é o nome do plugin)
│
├── .claude-plugin/
│   └── plugin.json                 ← A "carteira de identidade": nome, versão, descrição
│
├── hooks/                          ← AUTOMÁTICO: roda sozinho em certos eventos
│   ├── hooks.json                  ← a lista: "QUAL script roda em QUAL momento"
│   ├── lint-and-typecheck.sh       ← roda depois de cada edição (lint/type-check)
│   └── scope-cop.sh                ← roda antes de cada edição de UI (juiz de escopo)
│
└── skills/                         ← MANUAL: você invoca digitando o comando
    └── setup/
        └── SKILL.md                ← vira o comando /guardrails:setup
```

### O modelo mental: um plugin dá 2 tipos de coisa

1. **`hooks/`** = automático. Dispara sozinho em eventos (antes/depois de uma ação). Você não chama — ele intercepta. Os 3 guardrails são hooks.
2. **`skills/`** = manual. Você digita `/guardrails:setup` e ele roda. É um comando.

### As 3 regras de pasta que não são óbvias

- **`plugin.json` tem que estar dentro de `.claude-plugin/`** — não na raiz do plugin. É essa pasta que diz ao Claude "isto aqui é um plugin".
- **`hooks.json` tem que estar dentro de `hooks/`** — *nunca* na raiz do plugin. Se estiver na raiz, o Claude ignora em silêncio (o `claude plugin details` mostra `Hooks (0)`). É o gotcha #1 do teu repo.
- **Cada skill é uma subpasta de `skills/`** — o nome da pasta (`setup`) vira o nome do comando (`/guardrails:setup`).

### Por que separado de `hooks.json` os `.sh`?

`hooks.json` é só a **declaração** ("rode `scope-cop.sh` antes de Edit/Write"). Os `.sh` são o **código** que de fato roda. O `hooks.json` aponta pros scripts via `${CLAUDE_PLUGIN_ROOT}/hooks/scope-cop.sh` — uma variável que o Claude resolve pra onde quer que o plugin esteja instalado na máquina.
