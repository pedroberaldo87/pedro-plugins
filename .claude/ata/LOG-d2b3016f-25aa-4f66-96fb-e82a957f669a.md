## Decisão NOVA de hoje — R1 redefinido: o motor é um Workflow (não sub-agentes, não agent team)

O R1 do plano-fonte trocava as 6 lentes por "1 Opus Revisor + 1 Opus Planejador + Sonnet Executor" mas
**não cravava a mecânica**. Agora crava: **um Workflow** (a tool `Workflow`), porque o qa-loop é control-flow
**determinístico** (N rodadas, gate de severidade, churn detector, parada) com **agentes de modelos
diferentes por papel** — o caso canônico do Workflow.

Por que Workflow e não as outras 2 topologias:
- **vs sub-agente Task** — os agent() do Workflow são spawns, mas a orquestração é **código JS**, não o Opus
  decidindo spawnar Task à toa (o que a regra do Pedro condena). O gate/churn/parada viram `if`/contador/while,
  não "o Opus lembrar de aplicar a regra toda rodada". E não dispara o guard `PreToolUse(Agent)` a cada fix.
- **vs Agent Team** — agent team é pra colaboração aberta/bidirecional entre teammates nomeados; aqui o fluxo
  é um pipeline fechado (review→plan→exec→gate). Workflow dá determinismo + telemetria confiável + resumível;
  agent team deixaria a parada na cabeça do Opus.

**Fronteira motor (Workflow) × casca (skill) — a única concessão:** o Workflow roda em background e **não faz
AskUserQuestion no meio**. Logo, os 2 toques humanos do qa-loop ficam na **casca** (a skill, no Opus principal):

## Arquitetura do motor (o Workflow script)

A skill instrui o loop principal a invocar a tool `Workflow` com um script cujo `meta` declara as fases, e
cujo corpo é o loop de rodadas. Esqueleto (princípio, não código final):

- **Args do workflow** (passados pela casca): `target`, `planPath`, `severity_floor`, `max_rounds`,
  `domain`, `safety_layer` (a camada de rede detectada), e o que a casca leu do journal —
  `accepted_limits[]` + `invariants[]` (memória de QA do projeto) + `learnings` (afinam os prompts).
- **Loop de rodadas** (`while (!cleanRound && r < max_rounds && !churnEscalated)`):
  - **REVIEW** — `agent(reviewPrompt, {model:'opus', schema: FINDINGS})`. Um Opus Revisor **independente**
    cobrindo as 6 dimensões como **checklist de 1 agente** (arquitetura·backend·frontend·contratos·correção·UX).
    Rodada 1 = sweep completo; rodadas 2+ = caça-regressão nas mudanças + ângulos frescos; a rodada que
    confirma a parada re-roda o checklist completo. Recebe `accepted_limits` + `invariants` (não re-reportar /
    não violar). **Só acha — não julga severidade.**
  - **PLAN** — `agent(planPrompt(review, delta), {model:'opus', schema: PLAN})`. O Opus Planejador **adversarial**
    é o **árbitro único** (R2): adjudica "procede?" contra a rubrica, roteia nos 3 buckets (impl / plan-drift /
    plan-flaw), triagem por severidade **e** risco-de-conflito (mecanismos A/D), sequencia. Default a REJEITAR
    finding/accepted-limit sem justificativa. Rodada 1 = plano pesado; 2+ = só o DELTA.
  - **EXEC** — para cada finding do bucket 1, **em série** (o gate exige rodar a suíte entre fixes):
    `agent(execPrompt(fix), {model:'sonnet', schema: EXEC_RESULT})`. O Sonnet faz test-first (red) → fix
    cirúrgico (extensão enumerada > alargamento) → **roda a SUÍTE INTEIRA** → reporta. Nunca toca bucket 2/3.
  - **GATE de regressão (JS, não LLM)** — o `EXEC_RESULT` traz se a suíte regrediu (objetivo). Suíte vermelha →
    **reverte** em código; `churn[fn]++`. *Como* refazer (cirúrgico vs escalar) = mini-`agent(opus)` ou entra
    no PLAN incremental da próxima rodada. Segurança não delega ao Sonnet — quem decide keep/revert é Opus/JS.
  - **GATE de severidade (JS)** — `cleanRound = (review.findings.filter(sev≥floor && !inAcceptedLimits).length === 0)`
    sobre uma rodada **completa** (todo finder retornou). Churn por **função**: ≥`churn_threshold`(2) → escala.
  - acumula telemetria da rodada (`corrections`, `regressions`, `findings_by_sev`, `tokens`).
- **Return** — objeto estruturado: rodadas + findings + correções + **alertas plan-flaw** + accepted-limits
  propostos + telemetria. A casca consome isso pro relatório + journal.

Schemas (`FINDINGS`, `PLAN`, `EXEC_RESULT`) são o que torna o gate determinístico em JS — sem parsear texto.

## Decisão NOVA de hoje — R1 redefinido: o motor é um Workflow (não sub-agentes, não agent team)

O R1 do plano-fonte trocava as 6 lentes por "1 Opus Revisor + 1 Opus Planejador + Sonnet Executor" mas
**não cravava a mecânica**. Agora crava: **um Workflow** (a tool `Workflow`), porque o qa-loop é control-flow
**determinístico** (N rodadas, gate de severidade, churn detector, parada) com **agentes de modelos
diferentes por papel** — o caso canônico do Workflow.

Por que Workflow e não as outras 2 topologias:
- **vs sub-agente Task** — os agent() do Workflow são spawns, mas a orquestração é **código JS**, não o Opus
  decidindo spawnar Task à toa (o que a regra do Pedro condena). O gate/churn/parada viram `if`/contador/while,
  não "o Opus lembrar de aplicar a regra toda rodada". E não dispara o guard `PreToolUse(Agent)` a cada fix.
- **vs Agent Team** — agent team é pra colaboração aberta/bidirecional entre teammates nomeados; aqui o fluxo
  é um pipeline fechado (review→plan→exec→gate). Workflow dá determinismo + telemetria confiável + resumível;
  agent team deixaria a parada na cabeça do Opus.

**Fronteira motor (Workflow) × casca (skill) — a única concessão:** o Workflow roda em background e **não faz
AskUserQuestion no meio**. Logo, os 2 toques humanos do qa-loop ficam na **casca** (a skill, no Opus principal):
1. **Passo 0** — classificar domínio; se ambíguo e não-headless, pergunta 1× (`AskUserQuestion`).
2. **Passo 8** — veredito 0-10 do Pedro no fim (skipável).
Tudo entre esses dois pontos é o Workflow. Em modo `--headless` (/sovai) não há toque nenhum: a casca
auto-classifica, dispara o Workflow, e injeta o report no relatório do /sovai.

## Arquitetura do motor (o Workflow script)

A skill instrui o loop principal a invocar a tool `Workflow` com um script cujo `meta` declara as fases, e
cujo corpo é o loop de rodadas. Esqueleto (princípio, não código final):

- **Args do workflow** (passados pela casca): `target`, `planPath`, `severity_floor`, `max_rounds`,
  `domain`, `safety_layer` (a camada de rede detectada), e o que a casca leu do journal —
  `accepted_limits[]` + `invariants[]` (memória de QA do projeto) + `learnings` (afinam os prompts).
- **Loop de rodadas** (`while (!cleanRound && r < max_rounds && !churnEscalated)`):
  - **REVIEW** — `agent(reviewPrompt, {model:'opus', schema: FINDINGS})`. Um Opus Revisor **independente**
    cobrindo as 6 dimensões como **checklist de 1 agente** (arquitetura·backend·frontend·contratos·correção·UX).
    Rodada 1 = sweep completo; rodadas 2+ = caça-regressão nas mudanças + ângulos frescos; a rodada que
    confirma a parada re-roda o checklist completo. Recebe `accepted_limits` + `invariants` (não re-reportar /
    não violar). **Só acha — não julga severidade.**
  - **PLAN** — `agent(planPrompt(review, delta), {model:'opus', schema: PLAN})`. O Opus Planejador **adversarial**
    é o **árbitro único** (R2): adjudica "procede?" contra a rubrica, roteia nos 3 buckets (impl / plan-drift /
    plan-flaw), triagem por severidade **e** risco-de-conflito (mecanismos A/D), sequencia. Default a REJEITAR
    finding/accepted-limit sem justificativa. Rodada 1 = plano pesado; 2+ = só o DELTA.
  - **EXEC** — para cada finding do bucket 1, **em série** (o gate exige rodar a suíte entre fixes):
    `agent(execPrompt(fix), {model:'sonnet', schema: EXEC_RESULT})`. O Sonnet faz test-first (red) → fix
    cirúrgico (extensão enumerada > alargamento) → **roda a SUÍTE INTEIRA** → reporta. Nunca toca bucket 2/3.
  - **GATE de regressão (JS, não LLM)** — o `EXEC_RESULT` traz se a suíte regrediu (objetivo). Suíte vermelha →
    **reverte** em código; `churn[fn]++`. *Como* refazer (cirúrgico vs escalar) = mini-`agent(opus)` ou entra
    no PLAN incremental da próxima rodada. Segurança não delega ao Sonnet — quem decide keep/revert é Opus/JS.
  - **GATE de severidade (JS)** — `cleanRound = (review.findings.filter(sev≥floor && !inAcceptedLimits).length === 0)`
    sobre uma rodada **completa** (todo finder retornou). Churn por **função**: ≥`churn_threshold`(2) → escala.
  - acumula telemetria da rodada (`corrections`, `regressions`, `findings_by_sev`, `tokens`).
- **Return** — objeto estruturado: rodadas + findings + correções + **alertas plan-flaw** + accepted-limits
  propostos + telemetria. A casca consome isso pro relatório + journal.

Schemas (`FINDINGS`, `PLAN`, `EXEC_RESULT`) são o que torna o gate determinístico em JS — sem parsear texto.
