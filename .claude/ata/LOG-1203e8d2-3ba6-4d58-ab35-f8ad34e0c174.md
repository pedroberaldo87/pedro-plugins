2. **Plan-drift** (um "fix" otimizaria o código mas afastaria o comportamento do que foi pedido/planejado)

3. **Plan/arch flaw** (o plano em si é falho — decisão de arquitetura que gera problema crítico) →

## Arquitetura do loop (a versão do Pedro)

Domínio é uma **frase-guarda** no contrato, não um passo-0 pesado: "existe um comando objetivo pass/fail
(testes/build/lint)?" Se sim, esse comando vira um sinal de parada adicional (modo convergente embutido);
senão, governa o gate de severidade (assintótico, o caso primário).

Rodada 1 = REVIEW completo. Depois, o ciclo por rodada:

- **PLAN (formal, Opus, alto effort).** Onde moram os 3 buckets + anti-conflito + plan-fidelity. 4 sub-passos:
  (a) consolidar+dedup os findings (mecânica do rev6); (b) **rotular cada finding por bucket** (impl/drift/
  flaw); (c) pros de implementação: triar por severidade E por **risco de conflito** (3 sinais checáveis por
  leitura ANTES de editar — dois fixes na mesma função / fix que ALARGA regra genérica / fix que viola uma
  **invariante viva**), sequenciar (extensão-enumerada primeiro, alargamento por último com negative-tests,
  agrupar por função); (d) decidir por finding: conserta-agora | accepted-limit | escala-refator | **alerta**.
  Produz um artefato curto (tabela markdown + "invariantes vivas" + "descartados/aceitos/alertas").
  **Re-plano incremental:** rodada 1 = plano pesado; rodadas seguintes processam só o DELTA (o que a execução

- **EXEC (Sonnet, alto effort), só bucket 1, um fix por vez.** Para cada finding de implementação:

## CONSTRAINT CENTRAL — QA ancorado no plano + fronteira QA ≠ re-planejamento

A skill **não é só review de código** — audita **fidelidade ao plano** a cada rodada. Todo finding é
roteado em **3 buckets** no passo de planejamento:

1. **Implementação** (código diverge do plano / bug / regressão) → conserta no loop, com regression gate.
2. **Plan-drift** (um "fix" otimizaria o código mas afastaria o comportamento do que foi pedido/planejado)
   → **bloqueia/reverte**; o plano vence a "melhoria". Drift é uma **classe de regressão**, no mesmo gate
   que regressão de teste — mesmo que o agente ache que "faz sentido mudar".
3. **Plan/arch flaw** (o plano em si é falho — decisão de arquitetura que gera problema crítico) →
   **bucket de ALERTA**, nunca consertado/implementado no loop. Sobe pro Pedro no relatório:
   "apresento e julgamos". A skill **enforça** o plano; não o **redesenha**. "QA é QA."

Sinergia com um furo que a crítica achou: severidade atribuída por 6 LLMs oscila entre rodadas. Ancorar
em **"diverge do plano"** (objetivo, checável contra o doc) é mais estável que "é P1" (subjetivo) — a
constraint do Pedro conserta a fraqueza do critério.

## Arquitetura do loop (a versão do Pedro)

Domínio é uma **frase-guarda** no contrato, não um passo-0 pesado: "existe um comando objetivo pass/fail
(testes/build/lint)?" Se sim, esse comando vira um sinal de parada adicional (modo convergente embutido);
senão, governa o gate de severidade (assintótico, o caso primário).

Rodada 1 = REVIEW completo. Depois, o ciclo por rodada:

- **PLAN (formal, Opus, alto effort).** Onde moram os 3 buckets + anti-conflito + plan-fidelity. 4 sub-passos:
  (a) consolidar+dedup os findings (mecânica do rev6); (b) **rotular cada finding por bucket** (impl/drift/
  flaw); (c) pros de implementação: triar por severidade E por **risco de conflito** (3 sinais checáveis por
  leitura ANTES de editar — dois fixes na mesma função / fix que ALARGA regra genérica / fix que viola uma
  **invariante viva**), sequenciar (extensão-enumerada primeiro, alargamento por último com negative-tests,
  agrupar por função); (d) decidir por finding: conserta-agora | accepted-limit | escala-refator | **alerta**.
  Produz um artefato curto (tabela markdown + "invariantes vivas" + "descartados/aceitos/alertas").
  **Re-plano incremental:** rodada 1 = plano pesado; rodadas seguintes processam só o DELTA (o que a execução
  mudou + findings novos), não re-triam tudo — evita custo quadrático.
- **EXEC (Sonnet, alto effort), só bucket 1, um fix por vez.** Para cada finding de implementação:
  test-first (red) → fix cirúrgico (extensão enumerada > alargamento) → roda a SUÍTE INTEIRA → reporta.
  **Quem decide keep/revert é o Opus, não o Sonnet** — Sonnet aplica + roda + reporta; Opus decide
  manter / reverter+refazer cirúrgico / escalar (churn++). Segurança não delega. Sonnet **nunca** toca
  bucket 2/3 (não re-arquiteta).
- **REVIEW (sub-agentes exploram).** Rodada 1 = sweep completo (6 lentes rev6 em paralelo, mensagem única).
  Rodadas 2+ = 1 caça-regressão focado nas MUDANÇAS da rodada + 1-2 ângulos frescos (briefing: não re-varrer
  idêntico). **A rodada que CONFIRMA a parada re-roda o sweep cheio** — senão para numa rodada magra e perde
  ângulo (furo apontado pela crítica). Finder que não retornou = rodada incompleta, NUNCA "achou zero".
- **RE-PLAN** → volta ao PLAN incremental. Todos os finders recebem os **accepted-limits** vivos
  (não re-reportar) + as **invariantes vivas** (não violar).

Knobs explícitos e configuráveis: `planner_model` (default opus / high), `executor_model` (default
sonnet / high), `severity_floor` (default P1), `max_rounds` (teto-trava, default 6), `regression_gate` (on).

## Critério de parada (gate de severidade primário, teto = trava)

Predicado primário (assintótico), 3 cláusulas todas necessárias: **uma rodada INTEIRA completou
(todos os finders retornaram) E produziu ZERO findings novos de severidade ≥ floor (P0/P1) que estejam
FORA dos accepted-limits**. Quando ocorre, PARA — mesmo sem atingir o teto.

- **Teto (default 6) é SAFETY-CAP, não meta.** O texto da SKILL.md diz LITERALMENTE "trava de incêndio,
  não meta; quem decide é o gate de severidade". Teto disparar antes do gate = **alarme** ("não convergiu —
  aceita os limites ou refatora"), não sucesso. ⚠️ **DESVIO do briefing:** o briefing crava `max_rounds=3`;
  aqui é 6-como-trava + gate. Justificativa: cravar 3 é amostra n=1 (o erro simétrico ao "até zero"); o
  Pedro pediu coletar amostras antes (concern 1). A telemetria decide depois.
- **Modo convergente** (se há comando objetivo): para também quando o comando passa (exit 0) sem regressão
  lint/typecheck — a lógica do `/iterate`, agora dentro da `/qa-loop`.
- Dois riscos opostos cobertos: parar-cedo (perde bug) — enquanto houver P0/P1 fora dos limites NÃO para, e
  toda rodada seguinte tem 1 caça-regressão; picuinha-infinita — floor P1 descarta P2/P3, accepted-limits
  impede re-julgar, churn detector escala após 2 regressões na mesma função.

## Guard-rails operacionais (texto pronto entra na SKILL.md)

- **Regression gate por conserto:** test-first → fix → SUÍTE INTEIRA → regrediu? reverte + cirúrgico +
  churn++. É o maior ROI (converte 4 dos 5 mecanismos de "descoberto 3 rodadas depois" em "falha na hora").
- **5 mecanismos:** A (alargamento) → extensão enumerada > alargar regra; B (curto-circuito) → exceção
  específica DEPOIS da regra positiva, testar interseção; C (proxy sintático) → combinar sinais; D (conflito
  entre correções) → teste **nomeado com a invariante**, suíte inteira a cada conserto; E (interação entre
  estágios) → cada estágio neutraliza o que processou, testar a interação. A/D prevenidos no PLAN; B/C/E
  pegos pelo gate.
- **Sem suíte de testes (poda da crítica — 2 camadas, não 5):** tem suíte → roda o gate completo. **Não tem**
  → declara a cegueira em **linguagem humana** ("sem testes/lint/jornada, cada conserto é aposta cega") e
  **limita a 1 rodada de críticos**. Camada intermediária só se materializar: lint/typecheck como baseline
  (pega tipo/sintaxe, NÃO comportamento) ou jornada Playwright quando o alvo é UX. "Cumprir 100%" aqui =
  **honestidade 100%** sobre a rede, nunca segurança inventada. Detecção lint/typecheck/test = tabelas
  COPIADAS do `/iterate`+`/ship` (não referenciadas — `/iterate` vai ser apagado).
- **Churn detector** por **função** (não arquivo): ≥2 regressões auto-infligidas na mesma função → PARA de
  remendar, sinaliza "acoplamento alto: refatora pra detectores independentes OU aceita os limites".
- **Findings arquiteturais** (não viram teste-red): excluídos do gate, **só reportados** (viram bucket 3 ou
  P2 documentado) — nunca consertados sem rede.

## Relatório final + journaling

Relatório apresentado ao Pedro **e** persistido. Conteúdo: nº de loops + critério de parada que disparou;
**por loop** — correções aplicadas, **taxa de retorno** (findings novos de severidade real / yield),
**taxa de melhoria**, regressões pegas, custo (tokens); **curva de retornos decrescentes**; accepted-limits;
**bucket de alertas plan/arch-flaw** (apresentados, NÃO implementados — "apresento e julgamos").

Journaling (poda da crítica — enxuto, não 20 campos): os números **saem do relatório que a skill já gera**,
não é contabilidade extra. 1 linha por sessão em JSONL append-only com o essencial pra avaliar o nº ideal de
loops: `{target, domain, rounds_run, corrections_per_round, last_round_with_severe_finding, regressions,
tokens_per_round, stop_reason, pedro_score_0_10}`. Veredito 0-10 do Pedro = 1 pergunta opcional no fim
(âncora subjetiva). **Zero auto-tuning no v1** — o ajuste do teto é humano, informado por ≥10-15 sessões
(rejeição do loop autoresearch: **CONFIRMADO** na fonte real em disco que falta a métrica escalar comparável;
o keep/discard já existe no lugar certo = o regression gate, por teste objetivo).

**Onde mora o journal (recomendação, confirmar):** `.claude/qa-loop/` por projeto (markdown humano +
`telemetry.jsonl`, como o `/iterate` faz com `.claude/iterate/`) **+ cópia agregada no nível do usuário**
(`~/.claude/qa-loop/journal/`) que sobrevive a reinstalar o plugin e acumula cross-projeto — é o que viabiliza
"avaliação futura" do nº de loops. Adicionar `.claude/qa-loop/` ao **.gitignore**. **accepted-limits ancorado
ao alvo** (arquivo ao lado do código) pra sobreviver entre sessões (furo de estado apontado pela crítica).

## Arquitetura do loop (a versão do Pedro)

Domínio é uma **frase-guarda** no contrato, não um passo-0 pesado: "existe um comando objetivo pass/fail
(testes/build/lint)?" Se sim, esse comando vira um sinal de parada adicional (modo convergente embutido);
senão, governa o gate de severidade (assintótico, o caso primário).

Rodada 1 = REVIEW completo. Depois, o ciclo por rodada:

- **PLAN (formal, Opus, alto effort).** Onde moram os 3 buckets + anti-conflito + plan-fidelity. 4 sub-passos:
  (a) consolidar+dedup os findings (mecânica do rev6); (b) **rotular cada finding por bucket** (impl/drift/
  flaw); (c) pros de implementação: triar por severidade E por **risco de conflito** (3 sinais checáveis por
  leitura ANTES de editar — dois fixes na mesma função / fix que ALARGA regra genérica / fix que viola uma
  **invariante viva**), sequenciar (extensão-enumerada primeiro, alargamento por último com negative-tests,
  agrupar por função); (d) decidir por finding: conserta-agora | accepted-limit | escala-refator | **alerta**.
  Produz um artefato curto (tabela markdown + "invariantes vivas" + "descartados/aceitos/alertas").
  **Re-plano incremental:** rodada 1 = plano pesado; rodadas seguintes processam só o DELTA (o que a execução
  mudou + findings novos), não re-triam tudo — evita custo quadrático.
- **EXEC (Sonnet, alto effort), só bucket 1, um fix por vez.** Para cada finding de implementação:

  bucket 2/3 (não re-arquiteta).
- **REVIEW (sub-agentes exploram).** Rodada 1 = sweep completo (6 lentes rev6 em paralelo, mensagem única).

## Arquitetura do loop (a versão do Pedro)

Domínio é uma **frase-guarda** no contrato, não um passo-0 pesado: "existe um comando objetivo pass/fail
(testes/build/lint)?" Se sim, esse comando vira um sinal de parada adicional (modo convergente embutido);
senão, governa o gate de severidade (assintótico, o caso primário).

Rodada 1 = REVIEW completo. Depois, o ciclo por rodada:

- **PLAN (formal, Opus, alto effort).** Onde moram os 3 buckets + anti-conflito + plan-fidelity. 4 sub-passos:
  (a) consolidar+dedup os findings (mecânica do rev6); (b) **rotular cada finding por bucket** (impl/drift/
  flaw); (c) pros de implementação: triar por severidade E por **risco de conflito** (3 sinais checáveis por
  leitura ANTES de editar — dois fixes na mesma função / fix que ALARGA regra genérica / fix que viola uma
  **invariante viva**), sequenciar (extensão-enumerada primeiro, alargamento por último com negative-tests,
  agrupar por função); (d) decidir por finding: conserta-agora | accepted-limit | escala-refator | **alerta**.
  Produz um artefato curto (tabela markdown + "invariantes vivas" + "descartados/aceitos/alertas").
  **Re-plano incremental:** rodada 1 = plano pesado; rodadas seguintes processam só o DELTA (o que a execução
  mudou + findings novos), não re-triam tudo — evita custo quadrático.
- **EXEC (Sonnet, alto effort), só bucket 1, um fix por vez.** Para cada finding de implementação:
  test-first (red) → fix cirúrgico (extensão enumerada > alargamento) → roda a SUÍTE INTEIRA → reporta.
  **Quem decide keep/revert é o Opus, não o Sonnet** — Sonnet aplica + roda + reporta; Opus decide
  manter / reverter+refazer cirúrgico / escalar (churn++). Segurança não delega. Sonnet **nunca** toca
  bucket 2/3 (não re-arquiteta).
- **REVIEW (sub-agentes exploram).** Rodada 1 = sweep completo (6 lentes rev6 em paralelo, mensagem única).
  Rodadas 2+ = 1 caça-regressão focado nas MUDANÇAS da rodada + 1-2 ângulos frescos (briefing: não re-varrer
  idêntico). **A rodada que CONFIRMA a parada re-roda o sweep cheio** — senão para numa rodada magra e perde
  ângulo (furo apontado pela crítica). Finder que não retornou = rodada incompleta, NUNCA "achou zero".
- **RE-PLAN** → volta ao PLAN incremental. Todos os finders recebem os **accepted-limits** vivos
  (não re-reportar) + as **invariantes vivas** (não violar).

Knobs explícitos e configuráveis: `planner_model` (default opus / high), `executor_model` (default
sonnet / high), `severity_floor` (default P1), `max_rounds` (teto-trava, default 6), `regression_gate` (on).

## Critério de parada (gate de severidade primário, teto = trava)

Predicado primário (assintótico), 3 cláusulas todas necessárias: **uma rodada INTEIRA completou
(todos os finders retornaram) E produziu ZERO findings novos de severidade ≥ floor (P0/P1) que estejam
FORA dos accepted-limits**. Quando ocorre, PARA — mesmo sem atingir o teto.

- **Teto (default 6) é SAFETY-CAP, não meta.** O texto da SKILL.md diz LITERALMENTE "trava de incêndio,
  não meta; quem decide é o gate de severidade". Teto disparar antes do gate = **alarme** ("não convergiu —
  aceita os limites ou refatora"), não sucesso. ⚠️ **DESVIO do briefing:** o briefing crava `max_rounds=3`;
  aqui é 6-como-trava + gate. Justificativa: cravar 3 é amostra n=1 (o erro simétrico ao "até zero"); o
  Pedro pediu coletar amostras antes (concern 1). A telemetria decide depois.
- **Modo convergente** (se há comando objetivo): para também quando o comando passa (exit 0) sem regressão
  lint/typecheck — a lógica do `/iterate`, agora dentro da `/qa-loop`.
- Dois riscos opostos cobertos: parar-cedo (perde bug) — enquanto houver P0/P1 fora dos limites NÃO para, e
  toda rodada seguinte tem 1 caça-regressão; picuinha-infinita — floor P1 descarta P2/P3, accepted-limits
  impede re-julgar, churn detector escala após 2 regressões na mesma função.

## Guard-rails operacionais (texto pronto entra na SKILL.md)

- **Regression gate por conserto:** test-first → fix → SUÍTE INTEIRA → regrediu? reverte + cirúrgico +
  churn++. É o maior ROI (converte 4 dos 5 mecanismos de "descoberto 3 rodadas depois" em "falha na hora").
- **5 mecanismos:** A (alargamento) → extensão enumerada > alargar regra; B (curto-circuito) → exceção
  específica DEPOIS da regra positiva, testar interseção; C (proxy sintático) → combinar sinais; D (conflito
  entre correções) → teste **nomeado com a invariante**, suíte inteira a cada conserto; E (interação entre
  estágios) → cada estágio neutraliza o que processou, testar a interação. A/D prevenidos no PLAN; B/C/E
  pegos pelo gate.
- **Sem suíte de testes (poda da crítica — 2 camadas, não 5):** tem suíte → roda o gate completo. **Não tem**
  → declara a cegueira em **linguagem humana** ("sem testes/lint/jornada, cada conserto é aposta cega") e
  **limita a 1 rodada de críticos**. Camada intermediária só se materializar: lint/typecheck como baseline
  (pega tipo/sintaxe, NÃO comportamento) ou jornada Playwright quando o alvo é UX. "Cumprir 100%" aqui =
  **honestidade 100%** sobre a rede, nunca segurança inventada. Detecção lint/typecheck/test = tabelas
  COPIADAS do `/iterate`+`/ship` (não referenciadas — `/iterate` vai ser apagado).
- **Churn detector** por **função** (não arquivo): ≥2 regressões auto-infligidas na mesma função → PARA de
  remendar, sinaliza "acoplamento alto: refatora pra detectores independentes OU aceita os limites".
- **Findings arquiteturais** (não viram teste-red): excluídos do gate, **só reportados** (viram bucket 3 ou
  P2 documentado) — nunca consertados sem rede.

  leitura ANTES de editar — dois fixes na mesma função / fix que ALARGA regra genérica / fix que viola uma

  test-first (red) → fix cirúrgico (extensão enumerada > alargamento) → roda a SUÍTE INTEIRA → reporta.
  **Quem decide keep/revert é o Opus, não o Sonnet** — Sonnet aplica + roda + reporta; Opus decide
  manter / reverter+refazer cirúrgico / escalar (churn++). Segurança não delega. Sonnet **nunca** toca
  bucket 2/3 (não re-arquiteta).
- **REVIEW = 1 Opus REVISOR** (substituído por R1 — NÃO são 6 lentes). Um único Opus cobre as **6 dimensões
  como CHECKLIST** (arquitetura · backend · frontend · contratos fullstack · correção · UX). Rodada 1 = sweep
  completo do checklist; rodadas 2+ = foco nas MUDANÇAS (caça-regressão) + ângulos frescos do checklist (não
  re-varrer idêntico). **A rodada que CONFIRMA a parada re-roda o checklist completo** — senão para numa
  rodada magra. O Opus REVISOR só ACHA; quem julga a severidade é o Opus PLANEJADOR (R2, árbitro único).
- **RE-PLAN** → volta ao PLAN incremental. O Revisor da próxima rodada recebe os **accepted-limits** vivos
  (não re-reportar) + as **invariantes vivas** (não violar).

Knobs explícitos e configuráveis (via flags ou `.claude/qa-loop.config.md` — R3): `reviewer_model`
(default opus / high), `planner_model` (opus / high), `executor_model` (sonnet / high), `severity_floor`
(P1), `max_rounds` (teto-trava, 6), `churn_threshold` (2), `regression_gate` (on).

## Critério de parada (gate de severidade primário, teto = trava)

Predicado primário (assintótico), 3 cláusulas todas necessárias: **uma rodada INTEIRA completou
(todos os finders retornaram) E produziu ZERO findings novos de severidade ≥ floor (P0/P1) que estejam
FORA dos accepted-limits**. Quando ocorre, PARA — mesmo sem atingir o teto.

- **Teto (default 6) é SAFETY-CAP, não meta.** O texto da SKILL.md diz LITERALMENTE "trava de incêndio,
  não meta; quem decide é o gate de severidade". Teto disparar antes do gate = **alarme** ("não convergiu —
  aceita os limites ou refatora"), não sucesso. ⚠️ **DESVIO do briefing:** o briefing crava `max_rounds=3`;
  aqui é 6-como-trava + gate. Justificativa: cravar 3 é amostra n=1 (o erro simétrico ao "até zero"); o
  Pedro pediu coletar amostras antes (concern 1). A telemetria decide depois.
- **Modo convergente** (se há comando objetivo): para também quando o comando passa (exit 0) sem regressão
  lint/typecheck — a lógica do `/iterate`, agora dentro da `/qa-loop`.
- Dois riscos opostos cobertos: parar-cedo (perde bug) — enquanto houver P0/P1 fora dos limites NÃO para, e
  toda rodada seguinte tem 1 caça-regressão; picuinha-infinita — floor P1 descarta P2/P3, accepted-limits
  impede re-julgar, churn detector escala após 2 regressões na mesma função.

## Guard-rails operacionais (texto pronto entra na SKILL.md)

- **Regression gate por conserto:** test-first → fix → SUÍTE INTEIRA → regrediu? reverte + cirúrgico +
  churn++. É o maior ROI (converte 4 dos 5 mecanismos de "descoberto 3 rodadas depois" em "falha na hora").
- **5 mecanismos:** A (alargamento) → extensão enumerada > alargar regra; B (curto-circuito) → exceção
  específica DEPOIS da regra positiva, testar interseção; C (proxy sintático) → combinar sinais; D (conflito
  entre correções) → teste **nomeado com a invariante**, suíte inteira a cada conserto; E (interação entre
  estágios) → cada estágio neutraliza o que processou, testar a interação. A/D prevenidos no PLAN; B/C/E
  pegos pelo gate.
- **Sem suíte de testes (poda da crítica — 2 camadas, não 5):** tem suíte → roda o gate completo. **Não tem**
  → declara a cegueira em **linguagem humana** ("sem testes/lint/jornada, cada conserto é aposta cega") e
  **limita a 1 rodada de críticos**. Camada intermediária só se materializar: lint/typecheck como baseline
  (pega tipo/sintaxe, NÃO comportamento) ou jornada Playwright quando o alvo é UX. "Cumprir 100%" aqui =
  **honestidade 100%** sobre a rede, nunca segurança inventada. Detecção lint/typecheck/test = tabelas
  COPIADAS do `/iterate`+`/ship` (não referenciadas — `/iterate` vai ser apagado).
- **Churn detector** por **função** (não arquivo): ≥2 regressões auto-infligidas na mesma função → PARA de
  remendar, sinaliza "acoplamento alto: refatora pra detectores independentes OU aceita os limites".
- **Findings arquiteturais** (não viram teste-red): excluídos do gate, **só reportados** (viram bucket 3 ou
  P2 documentado) — nunca consertados sem rede.
